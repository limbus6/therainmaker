\# The M\&A Rainmaker — Naming and Worldbuilding Framework



\## 1. Purpose



This document defines the naming philosophy, naming rules, worldbuilding boundaries and procedural naming framework for \*\*The M\&A Rainmaker\*\*.



Its objective is to ensure that:



\* the game has a distinctive identity

\* names feel memorable and thematically rich

\* references to pop culture are coherent rather than random

\* the tone remains professional and elegant

\* procedural generation avoids repetition

\* implementation remains practical in HTML, CSS and vanilla JavaScript



This framework should be used whenever creating:



\* company names

\* private equity fund names

\* strategic buyer names

\* advisor firms

\* law firms

\* internal team names

\* client names

\* people names

\* project codenames

\* sector headlines

\* event labels

\* flavour text references



This file should be treated as the \*\*canonical naming and worldbuilding reference\*\* for all future content in the game.



---



\## 2. Worldbuilding Goal



\## 2.1 The intended tone



The world of \*\*The M\&A Rainmaker\*\* should feel like a recognisable professional M\&A environment with an underlying layer of stylised fictional identity.



The player should feel that:



\* the process is serious

\* the environment is corporate and credible

\* the names are more memorable than generic business-sim labels

\* the universe has flavour and personality

\* references reward attention without breaking immersion



The game should feel like a credible sell-side advisory simulation inhabiting a lightly mythologised corporate universe.



\## 2.2 Core principle



The game should use \*\*evocation, adaptation and stylised homage\*\*, not direct copy.



This means:



\* inspired by pop culture

\* not dependent on direct use of famous canon names

\* recognisable in flavour, not literal in execution



\## 2.3 Reference domains



Primary inspiration domains:



\* Star Wars

\* The Lord of the Rings

\* Harry Potter

\* Game of Thrones / A Song of Ice and Fire

\* X-Men

\* The Avengers / Marvel cosmic and heroic tone



Secondary adjacent inspiration allowed when tonally compatible:



\* Dune-style imperial and political naming

\* Foundation-style institutional naming

\* cyberpunk sci-fi corporate naming when restrained

\* mythic and occult language where corporate use remains plausible



\## 2.4 What the game must avoid



The game must avoid:



\* heavy parody

\* meme-level naming

\* direct lift of famous protected names as routine practice

\* juvenile humour in core corporate entities

\* overconcentration of one franchise in one deal unless intentionally themed

\* repeated use of the same roots across all entities

\* names that sound fake in a way that undermines deal credibility

\## 3. Design Philosophy



\## 3.1 The naming ideal



The ideal name should satisfy most of the following:



\* feels distinctive

\* sounds plausible in a business context

\* hints at myth, power, mystery, technology or prestige

\* has good visual presence in UI

\* is easy to read in inboxes and dashboards

\* can support repeated use across a game session without becoming irritating

\* is different enough from other names to reduce confusion



\## 3.2 The three-layer model



All naming should be built using a three-layer model.



\### Layer A — Functional readability



The name must be readable in a professional interface.



\### Layer B — Thematic flavour



The name should evoke fantasy, sci-fi, heroic, arcane or dynastic energy.



\### Layer C — World consistency



The name should fit the broader style of the deal, phase, sector and entities around it.



If a name succeeds at flavour but fails at readability or consistency, it should be rejected.



\## 3.3 The 70 / 20 / 10 rule



To preserve tonal discipline:



\* \*\*70%\*\* of names should feel plausible and professional first, referential second

\* \*\*20%\*\* may be clearly stylised or conspicuously allusive

\* \*\*10%\*\* may be overt easter eggs or stronger references



This ratio prevents the setting from becoming a costume party inside a deal room.

\## 4. Core Worldbuilding Logic



\## 4.1 Professional realism first



Whatever the naming style, the underlying world remains one of:



\* mandates

\* buyers

\* vendors

\* management teams

\* lawyers

\* DD advisers

\* teasers

\* IMs

\* models

\* process pressure

\* bidding dynamics



The pop culture inspiration should sit on top of this professional substrate.



\## 4.2 Fictional realism second



The universe is not a satire. It is a stylised business world.



Names may feel richer, sharper and more iconic than ordinary corporate naming, but the mechanics and business behaviour remain grounded.



\## 4.3 Controlled theatricality



The player should occasionally smile at a name or recognise an allusion, but never feel the game has stopped taking itself seriously.



This is especially important in:



\* client communications

\* offer comparison

\* legal negotiation

\* diligence issue handling

\* signing and closing moments



The joke must never be louder than the deal.

\## 10. Anti-Repetition Framework



Because the user explicitly wants an exhaustive framework that avoids repetition, the naming system must include formal anti-repetition rules.



\## 10.1 Why repetition is dangerous



Repetition creates:



\* confusion between entities

\* tonal flattening

\* procedural laziness

\* loss of memorability

\* UI readability problems

\* weak worldbuilding



If every second name contains “Vale”, “Raven” or “Nova”, the setting becomes mush.



\## 10.2 Hard repetition rules



For any single generated deal:



\* no root token should be used in more than \*\*2 major entity names\*\*

\* no exact full token combination should repeat

\* no two major counterparties in the same deal should start with the same prefix unless deliberately grouped

\* no more than \*\*1 visible direct easter-egg codename\*\* should appear in the same phase unless thematically intentional

\* if a deal uses a dominant aesthetic family, at least \*\*30%\*\* of names should still come from secondary pools to avoid monotony



\## 10.3 Soft repetition rules



The system should discourage:



\* three names in one inbox view starting with the same letter

\* repeated suffixes for all buyers, e.g. every fund ending in “Capital”

\* repeated surname patterns among key people

\* too many names sharing the same vowel rhythm or same length



\## 10.4 Name collision control



Before finalising a generated name, run checks for:



\* exact duplicate

\* same first token

\* same last token

\* same first and last letter pattern in same entity cluster

\* visual confusion in UI, for example Ravenhold vs Ravenhall vs Ravenshade



\## 10.5 Similarity scoring idea



Each generated name can receive a simple similarity score against already-used names.



Example dimensions:



\* shared prefix

\* shared root

\* shared suffix

\* same initial letter

\* same entity type

\* same length bucket



If score exceeds threshold, reroll.

\## 11. Naming Distribution Rules by Deal



Each deal should have a controlled naming palette.



\## 11.1 Primary aesthetic assignment



At deal generation, assign:



\* one primary aesthetic family

\* one secondary aesthetic family

\* one tertiary reserve family



Example:



```js

{

&nbsp; primaryStyle: "space\_opera\_corporate",

&nbsp; secondaryStyle: "heroic\_modern",

&nbsp; tertiaryStyle: "arcane\_academic"

}

```



\## 11.2 Distribution example



For a tech-enabled industrial asset:



\* 50% primary style

\* 30% secondary style

\* 20% tertiary style



This gives coherence without uniformity.

\## 12.1 What counts as good flavour



Good flavour:



\* Palantir Advisory

\* Hyperion Dynamics

\* Ravenhold Capital

\* Elena Blackmere

\* Project Sentinel



These are evocative and usable.

Too direct:



\* Stark Industries

\* Hogwarts Ventures

\* Darth Vader Capital

\* House Targaryen Equity

\* Gandalf Advisory

These collapse the game into reference comedy.



\## 12.3 What counts as too silly



Too silly:



\* Death Star Debt Fund

\* Mordor Mid-Market Partners

\* Thanos Growth Capital

\* Hufflepuff Infrastructure plc



These might amuse briefly and then shred the atmosphere.



\## 12.4 Strong references are still allowed in controlled areas



The following are safer places for stronger allusion:



\* project codenames

\* occasional headline puns

\* rare easter-egg buyer alias in non-core content

\* achievement names

\* hidden flavour text



The following should remain more restrained:



\* major client names

\* senior advisor identity

\* law firms

\* final SPA counterparties

\## 15. Procedural Generation Logic



\## 15.1 Generation philosophy



Procedural naming should combine:



\* aesthetic family selection

\* entity type rules

\* token pools

\* repetition control

\* readability checks

\* reserved name lists



\## 15.2 Recommended generation sequence



1\. Select entity type

2\. Select style family based on deal distribution

3\. Select structure pattern

4\. Draw prefix / root / suffix tokens

5\. Check duplication and similarity

6\. Apply role-specific plausibility filter

7\. Store used tokens and reserved combinations



\## 15.3 Example company generation patterns



Pattern A:



```text

Prefix + Root + Suffix

Silver + mere + Holdings = Silvermere Holdings

```



Pattern B:



```text

Prefix + Root

Hyperion + Dynamics = Hyperion Dynamics

```



Pattern C:



```text

Compound Prefix + Suffix

Northspire + Capital = Northspire Capital

```



Pattern D:



```text

Two-word elevated name + suffix

Obsidian Arc + Partners = Obsidian Arc Partners

```

\## 16.4 Similarity filter example



```js

function isTooSimilar(candidate, existingNames) {

&nbsp; const c = candidate.toLowerCase();

&nbsp; return existingNames.some(name => {

&nbsp;   const n = name.toLowerCase();

&nbsp;   const sameStart = c.slice(0, 4) === n.slice(0, 4);

&nbsp;   const sameEnd = c.slice(-4) === n.slice(-4);

&nbsp;   const sameLengthBucket = Math.abs(c.length - n.length) <= 2;

&nbsp;   return (sameStart \&\& sameEnd) || (sameStart \&\& sameLengthBucket);

&nbsp; });

}

```



This is simple but useful for prototype stage.



\## 16.5 Name generation pseudocode



```js

function generateCompanyName(entityType, style, state) {

&nbsp; for (let i = 0; i < 50; i++) {

&nbsp;   const pattern = pickPattern(entityType, style);

&nbsp;   const candidate = buildNameFromPattern(pattern, style);

&nbsp;   if (!passesEntityRules(candidate, entityType)) continue;

&nbsp;   if (state.usedNames.companies.has(candidate)) continue;

&nbsp;   if (isTooSimilar(candidate, \[...state.usedNames.companies])) continue;

&nbsp;   if (!passesTokenLimits(candidate, state.usedNames.tokens)) continue;

&nbsp;   registerName(candidate, "companies", state);

&nbsp;   return candidate;

&nbsp; }

&nbsp; return fallbackCompanyName(entityType, style, state);

}

```

\## 17. Manual Curation Rules



Even with procedural generation, some manual curation should be used for:



\* flagship deals

\* named tutorial scenarios

\* key rivals

\* recurring advisors

\* marquee PE houses

\* top-tier strategic buyers



These named entities should become part of the game’s identity.



\## 17.1 Recurring signature entities



The game can benefit from a small set of recurring marquee names, but they must be limited.



Suggested frequency rules:



\* 3 to 5 recurring PE houses maximum

\* 3 to 5 recurring law firms maximum

\* 4 to 8 recurring strategic buyers maximum by macro-sector

\* all else primarily procedural



This prevents overexposure while preserving recognisable rivals and institutions.

\## 21. Canonical Naming Workflow for Content Designers



Whenever adding a new name manually, use this checklist.



1\. What entity type is this.

2\. What sector and deal style does it belong to.

3\. Which aesthetic family is primary here.

4\. Does the name sound credible in a professional setting.

5\. Is the reference evocative rather than literal.

6\. Does the name differ enough from existing active names.

7\. Will it remain readable in inbox and dashboard view.

8\. Does it fit the deal’s tonal distribution.

9\. Is it too funny for a serious moment.

10\. Would the name still feel good after seeing it twenty times.



If the answer to the last question is no, discard it.



---



\## 22. Canonical Naming Rules Summary



1\. Use homage, not direct copying.

2\. Preserve a professional M\&A tone at all times.

3\. Distinguish entity types through different naming styles.

4\. Use aesthetic families to create coherence.

5\. Control repetition through token tracking and similarity filtering.

6\. Allow stronger references mainly in project codenames and hidden flavour content.

7\. Keep law firms, formal counterparties and legal contexts more grounded.

8\. Use recurring marquee entities sparingly.

9\. Ensure names remain readable and UI-friendly.

10\. Never let naming become louder than the deal itself.

\## 24. Final Note



Naming is not decorative.



In a simulation game like \*\*The M\&A Rainmaker\*\*, names carry:



\* memory

\* atmosphere

\* differentiation

\* perceived world depth

\* emotional texture

\* replayability value



Generic names make the world feel disposable.



Over-literal names make it feel silly.



A disciplined pop-culture-inspired naming framework creates the sweet spot: a corporate simulation universe that feels sharp, memorable and lightly mythic without losing its professional spine.



That balance is the whole trick.
