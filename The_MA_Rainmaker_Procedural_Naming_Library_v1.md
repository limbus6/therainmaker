\# The M\&A Rainmaker — Procedural Naming Library v1



\## 1. Purpose



This document defines the expanded procedural naming library for \*\*The M\&A Rainmaker\*\*.



Its purpose is to provide a deep, reusable, low-repetition bank of naming material for:



\* client companies

\* strategic buyers

\* private equity funds

\* advisors

\* law firms

\* lenders

\* media outlets

\* internal units

\* project codenames

\* people names

\* event labels

\* achievement labels



This file should be used together with:



\* \*\*The\_MA\_Rainmaker\_Naming\_and\_Worldbuilding\_Framework.md\*\*

\* future procedural generation rules in vanilla JavaScript



The objective is not just to generate names. The objective is to generate names that remain:



\* distinctive

\* readable

\* atmosphere-rich

\* procedurally sustainable

\* controlled against repetition



---



\## 2. Library Design Rules



\## 2.1 Composition logic



Names should be built from modular layers:



\* style family

\* prefix pool

\* root pool

\* suffix pool

\* entity-type constraints

\* rarity rules

\* anti-repetition rules



\## 2.2 Token categories



Tokens are organised into:



\* common

\* uncommon

\* rare

\* marquee



\### Common



Stable, readable, broadly usable tokens.



\### Uncommon



More distinctive, used to enrich variety.



\### Rare



Should appear occasionally to preserve freshness.



\### Marquee



Reserved for signature names, premium firms, tutorial scenarios, flagship deals or rare codename drops.



\## 2.3 Entity-type discipline



Not every token should be available for every entity class.



For example:



\* “Quill” is good for advisors and media, less good for industrial manufacturers

\* “Forge” is excellent for industrial and heroic-tech companies, less good for law firms

\* “Citadel” is strong for PE, lenders and strategic buyers, usually too heavy for mid-market service firms

\## 8. Rarity and Weighting Rules



\## 8.1 Suggested weighting by category



\### Common



Weight 50



\### Uncommon



Weight 30



\### Rare



Weight 15



\### Marquee



Weight 5



\## 8.2 Entity weighting adjustments



For marquee entities:



\* increase uncommon and rare token chance

\* allow selected marquee tokens



For standard generated entities:



\* rely mostly on common and uncommon pools

\* use rare pools occasionally

\* reserve marquee pools unless scenario logic elevates entity status



\## 8.3 Anti-fatigue rule



If a token family has appeared prominently in the previous two deals, reduce its weight for the next generated deal unless manually overridden.

\## 9. Anti-Repetition Procedures



\## 9.1 Token caps per deal



Suggested hard caps:



\* prefix token max 2 uses per deal for major entities

\* root token max 2 uses per deal for major entities

\* suffix token max 4 uses per deal, with variation across entity classes

\* marquee token max 1 use per deal unless built into tutorial content



\## 9.2 UI view anti-clustering



In a single inbox or shortlist screen:



\* no more than 2 visible company names beginning with the same first 3 letters

\* no more than 2 surnames ending with the same final 4 letters

\* avoid three or more buyers with identical suffix class, e.g. all ending in Capital



\## 9.3 Cross-run anti-fatigue



Maintain a persistent history of recently used:



\* codenames

\* premium prefixes

\* recurring buyer names

\* recurring executive surnames



Then deprioritise recently used items.

\## 13. Final Rule Summary



1\. Generate names by entity type, not from one universal soup.

2\. Use rarity and marquee rules to preserve freshness.

3\. Track token use and similarity aggressively.

4\. Reserve premium names for premium moments.

5\. Make names readable in UI before making them clever.

6\. Blend aesthetic coherence with controlled variation.

7\. Avoid direct franchise copying.

8\. Prefer memorable plausibility over obvious reference comedy.
